## 目标

在当前视频网站项目基础上继续演进，补齐即时通信、认证增强、工程化与文档化能力。本次作业不再是单纯补接口，而是强调：

- 在原项目上持续迭代，而不是推倒重写
- 提升项目结构、代码复用性与配置管理能力
- 从单体项目逐步演进到微服务架构
- 引入服务注册与发现、RPC 调度与服务治理能力
- 引入 WebSocket、MFA、CI、参数校验、流量治理等现代后端要素
- 保留可答辩、可部署、可展示的工程交付物

## 目的

- 掌握 HTTP 协议和 Web 工作原理
- 掌握 WebSocket 原理与实践
- 掌握关系型数据库的基本操作
- 掌握微服务架构和 RPC 调度方法
- 完成项目结构优化
- 完成一定的自动化流程
- 完善上一个项目的文档

## 背景

FanOne 现在不仅要看视频，还要进行站内即时聊天。本次作业要求你在现有项目基础上扩展出 IM 能力，并同步补齐安全、配置、测试、CI 与项目文档；同时按 2025 版要求，将系统进一步演进为可答辩的微服务架构。

## 微服务主线要求

2025 版要求明确提出：本轮项目需要采用**微服务架构**，并重点完成以下内容：

1. 使用微服务思路重组现有系统，而不是继续维持单体应用。
2. 需要体现 HTTP 与 RPC 的职责分离。
3. 需要实现服务注册与发现。
4. 需要在报告中对比单体架构与微服务架构的优缺点。

推荐执行口径：

- 对外保留统一 API 网关或 BFF 层，处理 HTTP / 鉴权 / 参数绑定。
- 对内按领域拆分 RPC 服务，例如用户、视频、互动、聊天。
- 优先选用 Kitex 完成内部 RPC 通信，以贴近 2025 文档要求。
- 服务注册与发现可结合 etcd / Nacos / Consul 等方案，重点是“动态发现”而不是静态硬编码地址。

## 新增最低接口

本次新增 4 个最低要求接口：

| 模块 | 最低需要完成的接口 | 数量 |
| --- | --- | --- |
| 用户 | 获取 MFA 二维码、绑定 MFA | 2 |
| 视频 | 视频流 | 1 |
| 互动 | 无新增独立接口 | 0 |
| 社交 | 聊天 | 1 |

说明：

- 聊天功能要求基于 WebSocket 实现。
- Hertz 项目优先使用框架内置 WebSocket 能力。
- 视频模块新增视频流能力，需满足视频在线播放的基本场景。

## 两套要求的合并口径

官方文档存在“简单提升版”和“2025 微服务版”两条要求线。对当前仓库，建议按更高标准统一执行：

1. 保留 Lab4 既有核心接口能力，不能因重构为微服务而退化。
2. 补齐简单提升版中的新增能力：MFA、视频流、聊天、评论回复、评论点赞。
3. 同时满足 2025 微服务版要求：微服务拆分、Kitex、服务注册与发现、报告中的架构对比分析。

因此，本仓库默认采用“全部做高标准版”的执行口径，而不是只满足其中一条文档。

## 必须补齐的既有能力

虽然互动模块没有新增独立接口数量，但以下能力仍需从现有“简化版”升级为“完整支持”：

1. 评论接口需要支持“对评论进行评论”，即正确处理 `comment_id` 字段。
2. 点赞接口需要支持“对评论点赞”，不再只处理视频点赞。
3. 社交聊天需要基于 `Redis + MySQL` 实现，兼顾实时消息与持久化。

## 聊天与 AI 扩展要求

社交模块聊天功能除基础用户通信外，还要求考虑 AI 接入：

1. 用户对话过程中，AI 需要能够参与聊天。
2. AI 回复可以采用简化策略触发，例如收到消息即触发，或由服务端规则判断是否回复。
3. 需要为 AI 接入 `tool call` 能力。
4. 服务端需要具备 MCP 接入能力，并将其纳入 AI 的工具调用链路。
5. 若接入福 uu 已提供的 MCP tool，需要补充一个接口，让用户传入账号密码，并通过 `jwch` 获取 `id` 与 `cookie` 后传入工具侧。

工程建议：

- 优先参考成熟实现，不建议从零手搓完整 Agent Loop。
- 可参考 Bifrost、Pi、Claude Code 开源实现的思路。
- 当前阶段以“跑通基本能力 + 可解释设计方案”为主，不必盲目追求复杂多 Agent 架构。

## 项目结构要求

目录结构需要继续优化，重点考察“他人是否容易理解你的项目”。

要求：

1. 如果当前目录结构仍有明显混乱之处，需要继续调整。
2. README 或文档中提供目录树。
3. 目录树建议使用 `tree` 命令生成，不要求精确到每一个文件，但要覆盖关键目录和关键文件。
4. 若改造成微服务，目录结构需要明确区分：
   - 网关层 / HTTP 接入层
   - 各领域 RPC 服务
   - 公共 proto / idl
   - 服务发现与公共基础设施配置

## 微服务拆分要求

这是当前仓库后续开发的重点。

最低建议拆分如下：

1. `gateway`：统一 HTTP 接入，负责鉴权、参数校验、协议转换、Swagger 暴露。
2. `user-service`：用户、登录、头像、MFA。
3. `video-service`：投稿、列表、搜索、视频流、热榜相关读取。
4. `interaction-service`：点赞、评论、评论回复、评论点赞。
5. `chat-service`：WebSocket 聊天、会话、消息持久化、AI 消息协同。

要求：

1. 服务边界需要尽量按领域划分，避免“伪微服务”。
2. 服务间调用优先使用 RPC，而不是继续直接跨模块调用本地函数。
3. 共享内容通过公共 proto、公共包或基础设施层复用，不要重新耦合成单体。

## 服务注册与发现要求

服务注册与发现是 2025 文档里的明确要求，不能省略。

最低要求：

1. 服务启动时完成注册。
2. 服务消费者通过注册中心动态发现目标服务地址。
3. 不允许在生产链路中硬编码服务实例地址作为唯一方案。
4. 文档中说明注册中心选型、注册流程、发现流程与故障处理策略。

建议交付内容：

- 注册中心选型说明，例如 etcd。
- 服务注册流程图。
- 服务调用链示意图。
- 本地启动与 Docker 部署方式。

## RPC 与框架要求

2025 版文档明确点名 **Kitex**。对当前仓库，建议采用：

1. 对外 HTTP：保留 Hertz 作为网关层。
2. 对内 RPC：使用 Kitex 进行服务间通信。
3. 公共协议：继续以 Protobuf / IDL 统一维护。

这样可以同时兼顾：

- 现有 Hertz 项目基础
- 2025 微服务版对 Kitex / RPC 的要求
- WebSocket 聊天接入的落地便利性

## 代码复用性要求

本次作业明确要求关注代码复用。

要求：

1. 对重复出现的逻辑进行提取、整合或抽象。
2. 对常见公共逻辑避免散落在多个 handler / service / dao 中重复实现。
3. 需要在文档中说明本轮做了哪些复用性优化；如果认为没有明显可优化空间，也应明确说明判断依据。

## 细节优化要求

### 错误处理

- 对照推荐文章检查错误处理是否合理。
- 避免吞错、重复包装无效错误、日志与返回信息不一致等问题。

### 参数校验

- Hertz 项目需要接入参数绑定与校验能力。
- 所有核心接口至少应保证关键字段校验齐全。

### 流量治理

- Hertz 项目需集成 Sentinel。
- 要求支持自定义配置，而不是只停留在默认接入。

### 代码生成

- 如果上一轮没有使用 `hz` 或 `goctl`，这一轮必须改为使用代码生成工具。

### 常量规范

- 统一管理重复数字、文案、状态值、错误文本、缓存 key 前缀等常量。
- 降低散落硬编码带来的维护成本。

### 配置管理

新增强制要求：

1. 独立 `config/` 目录
2. 提供 `config.yaml`
3. 不再使用 ini
4. 配置需支持热更新
5. 推荐使用 Viper
6. `config/sql/` 目录下保存数据库建表语句，如 `init.sql`
7. 如果采用微服务，需要说明哪些配置是服务级配置，哪些是公共基础设施配置
